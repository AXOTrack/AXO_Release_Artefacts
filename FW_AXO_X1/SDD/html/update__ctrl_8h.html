Detailed Description
This header file defines the update controller module for managing firmware updates in the AXO X1 device. The update controller handles the logic for handling OTA commands, firmware update confirmation/rejection, and managing the retry mechanism for firmware downloads. It utilizes the Finite State Machine (FSM) for coordinating different stages of the update process.
An OTA_START command received via MQTT triggers the update mechanism. The update control task transitions to state triggered, subscribes to topic device/${device_id}/ota/update and waits for firmware information. On firmware info received (URL, filename, SHA1, etc.) it changes to state download and starts downloading the new image via HTTP to the secondary partition of internal flash. On successful download, the task changes to state ready and publishes a success message via MQTT. After a timeout to allow the MQTT message to be sent, the task triggers a reboot of the device. On reboot, the boot-loader MCUBoot (not part of the update container) performs the actual application firmware update by swapping the two images in slot #1 and slot #2 of internal flash. The application is always started from the image in slot #1. If the application does not mark the copied image as “confirmed” on first run, then on next reboot MCUBoot would revert the update by swapping the images a second time.

See also message sequence chart in section Firmware Update (FOTA) and statemachine described in class FwFsm
