Firmware update control task class derived from task base class.

Detailed Description
The firmware update control task implements the logic for handling OTA commands and firmware update confirmation or rejection. Furthermore it manages the retry mechanism for firmware downloads in case of an error. The class provides a Finite State Machine (FSM) for coordinating different stages of the update process, a timer used for delay reboot and download retry interval and functions to start the timer for the different purposes. It provides a function to request a download retry and functions to send JSON strings to the MQTT broker for confirmation and rejection.

◆ task()

void task ( void  )
virtual

Firmware update control task function.

The function directly enters the endless loop and wait for messages. Task wakes up when a message is in the mailbox and read the message by calling mailbox read method which returns the pointer pElem to the message. All expected messages will be handled, in case of an unexpected message an error debug message will be triggered. Finally the message will be released by calling class PoolElem method free, if the pointer pElem is not explicitly set to 0.

Message Handling:

  • eTimerExpired: The pointer pElem will be set to 0, to prevent releasing the message because it is allocated static. Afterwards handle_timer_exp method of the state machine instance is called.
  • eOtaStart: Method handle_OtaStart of the state machine instance is called.
  • eFwInfo: Method handle_FwInfo of the state machine instance is called.
  • eFwDlSuccess: Method handle_DlComplete of the state machine instance is called.
  • eFwDlFailed: Method handle_DlFailed of the state machine instance is called.

◆ startRetryTimer()

void startRetryTimer ( )

Function to start one-shot retry timer.

The function checks the retry counter, if it is the first retry timeout value is set to initial retry interval, otherwise current retry timeout will be multipied by 2. If current retry timeout is higher than maximum retry interval it will be limited to maximum retry interval. Finally the one-shot retry timer will be started with a duration of current retry timeout and the retry counter will be incremented by one.
